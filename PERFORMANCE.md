# 성능/토큰 최적화 규칙

CLAUDE.md와 Claude Code 기본 동작에 이미 있는 내용은 여기 반복하지 않는다. 아래는 이 프로젝트 고유의 추가 규칙만 담는다.

## 검증 실행 전략 (빠른 것부터, 필요한 것만)

- 코드 변경 후 기본 검증은 **typecheck + 변경 파일의 타깃 테스트**(`pnpm test <path>`)로 충분한지 먼저 판단한다. 전체 `pnpm test`는 커밋 직전 최종 확인에서만.
- `pnpm build`는 라우팅·설정(`next.config`, `tsconfig`, MSW 초기화 등)을 건드렸을 때만 실행한다. 컴포넌트/로직 변경에는 돌리지 않는다.
- `pnpm test:e2e`는 사용자 플로우가 실제로 바뀐 경우에만. 매 변경마다 돌리지 않는다.
- 독립적인 검증(lint + typecheck + test)은 quick-check 서브에이전트로 병렬 실행한다.

## 긴 파일 읽기

- `src/mocks/handlers.ts`와 `src/mocks/persisted-store.ts`는 계속 길어지는 파일이다. 전체를 읽지 말고 Grep으로 해당 엔드포인트/핸들러를 찾아 그 범위만 `offset`/`limit`으로 읽는다.
- 큰 페이지 컴포넌트(`*-page.tsx`)와 그 테스트도 동일 — 수정할 섹션만 읽는다.

## 코드 작성 시 성능 기본값

- React 19 / Next 16: 상호작용이 없으면 서버 컴포넌트 유지. `"use client"`는 이벤트 핸들러·훅이 실제로 필요한 최소 단위 컴포넌트에만 붙인다.
- Tailwind v4 유틸리티로 해결 가능한 스타일에 커스텀 CSS를 추가하지 않는다.
- 새 기능의 데이터 접근은 반드시 feature의 `api/`를 통해서만 — 컴포넌트가 MSW 엔드포인트 URL을 직접 알게 하지 않는다 (백엔드 교체 시 수정 범위 최소화).

## 조사 위임

- 3회 이상 Grep/Glob이 필요할 것 같은 개방형 코드 조사는 Explore 서브에이전트에 위임하고, 메인 세션에는 결론만 가져온다.
